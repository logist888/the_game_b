# Бэкапы и откат «Вавилона»

Деплой идёт автоматически при пуше в `main` (GitHub Pages). Чтобы каждый
релиз можно было откатить, действует двухуровневая страховка.

## 1. Автоматические точки восстановления (теги)

Workflow `.github/workflows/backup.yml` на **каждый пуш в `main`** создаёт тег
вида `backup/2026-06-16-070000-ab12cd3`, указывающий на этот коммит. То есть
любая задеплоенная версия остаётся восстановимой.

Посмотреть все точки: вкладка **Tags** на GitHub
(`https://github.com/logist888/the_game_b/tags`) или:

```bash
git fetch --tags
git tag -l 'backup/*' | sort
```

## 2. Ручная бэкап-ветка перед крупными мёржами

Перед мёржем большой фичи снимается ветка-снимок текущего `main`, например
`backup/v25-stable`. Ветки видны в списке веток репозитория.

## Как откатить прод на предыдущую версию

1. Выбери тег/ветку нужной рабочей версии (например `backup/2026-06-16-070000-ab12cd3`).
2. Верни `main` на неё и запушь:

```bash
git fetch origin --tags
git checkout main
git reset --hard backup/2026-06-16-070000-ab12cd3   # или backup/v25-stable
git push --force origin main
```

После форс-пуша GitHub Pages пере-задеплоит откатанную версию (1–2 минуты).

> ⚠️ `--force` перезаписывает историю `main`. Это и есть механизм отката —
> применяй осознанно. Новые точки-бэкапы при этом не теряются (они в тегах).

### Безопаснее: откат через revert (без перезаписи истории)

Если нужно отменить один проблемный мёрж, не трогая остальное:

```bash
git checkout main && git pull
git revert -m 1 <sha-merge-коммита>   # -m 1 для merge-коммита
git push origin main
```
